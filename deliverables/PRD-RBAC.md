# 增量PRD：多角色权限体系（RBAC）

## 一、产品目标

为精益成本管理系统添加完整的角色权限管理能力，使系统能够支撑企业级多用户协作场景，确保不同岗位人员只能访问其职责范围内的数据和功能。

### 目标用户画像
- **年产值1000万~5亿的制造型中小企业**
- 老板/总经理需要全局视角和审批权限
- 各部门主管需要本部门模块的管理权限
- 普通员工只能查看和操作本职相关内容

## 二、现有系统状态

### 当前用户模型
- `users` 表字段：id, username, password_hash, real_name, role, department, phone, avatar_url, status, created_at, updated_at
- `role` 字段当前值：`admin`（管理员）、`manager`（经理）、`staff`（员工）—— 但实际只做了占位，**没有任何权限控制逻辑**
- 后端 `requireRole()` 中间件已存在但**所有路由都未使用**
- 前端所有菜单对所有用户完全可见，路由无权限守卫

### 现有模块（1+6+1）
1. 成本驾驶舱（Dashboard）
2. 成本核算（Cost Accounting）：核算总览、材料成本、订单追溯
3. 采购管理（Procurement）：供应商管理、TCO分析、价格趋势
4. 库存控制（Inventory）：ABC分析、安全库存、呆滞预警
5. 生产精益（Production）：OEE计算、价值流图、SMED换模
6. 质量成本（Quality）：质量成本总览、PAF分析、SPC控制图
7. 激励中心（Incentive）：分享计算器、改善提案、积分排行榜

## 三、角色定义

### 角色层级

| 角色代码 | 角色名称 | 典型用户 | 描述 |
|---------|---------|---------|------|
| `super_admin` | 超级管理员 | 系统运维/IT | 管理所有租户、系统配置、用户管理 |
| `owner` | 企业老板/总经理 | 企业决策者 | 全模块读写、审批权、激励规则配置 |
| `finance_manager` | 财务经理 | 财务部门负责人 | 成本核算全权限、驾驶舱、质量成本、库存只读 |
| `procurement_manager` | 采购主管 | 采购部门负责人 | 采购管理全权限、库存读写、成本核算只读 |
| `production_manager` | 生产主管 | 生产部门负责人 | 生产精益全权限、质量成本读写、库存只读 |
| `warehouse_manager` | 仓管员 | 仓库人员 | 库存控制全权限、其他模块只读 |
| `quality_manager` | 质量主管 | 质量部门负责人 | 质量成本全权限、生产只读、库存只读 |
| `staff` | 普通员工 | 各部门员工 | 驾驶舱只读、改善提案提交、其他受限 |

### 权限矩阵（R=读写, r=只读, -=不可见）

| 模块 | super_admin | owner | finance | procurement | production | warehouse | quality | staff |
|------|:-----------:|:-----:|:-------:|:-----------:|:----------:|:---------:|:-------:|:-----:|
| 成本驾驶舱 | R | R | R | r | r | r | r | r |
| 成本核算 | R | R | R | r | - | - | - | - |
| 采购管理 | R | R | r | R | - | r | - | - |
| 库存控制 | R | R | r | R | r | R | r | - |
| 生产精益 | R | R | - | - | R | - | r | - |
| 质量成本 | R | R | R | - | r | - | R | - |
| 激励中心 | R | R | r | r | r | r | r | r* |
| 用户管理 | R | R | - | - | - | - | - | - |

> *staff在激励中心只能提交改善提案和查看排行榜，不能使用分享计算器

## 四、用户故事

### US-1 超级管理员管理用户
> 作为超级管理员，我希望能在后台新增/编辑/禁用用户并分配角色，以便控制谁可以访问系统的哪些功能。

**验收标准：**
- 用户列表页面展示所有用户（用户名、姓名、角色、部门、状态）
- 新增用户时必填：用户名、密码、姓名、角色
- 编辑用户时可修改：姓名、角色、部门、手机、状态
- 禁用用户后该用户无法登录
- 超级管理员不能禁用自己

### US-2 企业老板全局管控
> 作为企业老板/总经理，我希望登录后看到全部模块和所有数据，并能够审批改善提案和配置激励规则，以便掌控全局成本状况。

**验收标准：**
- 登录后侧边栏展示所有模块菜单
- 所有页面数据读写无限制
- 可审批改善提案（通过/驳回）
- 可修改激励规则

### US-3 部门主管限定模块
> 作为采购主管，我希望登录后只看到采购管理和库存相关模块，其他模块不显示，以便专注本职工作。

**验收标准：**
- 侧边栏只显示有权限的模块菜单
- 直接输入URL访问无权限页面时，显示403提示
- 后端API对无权限请求返回403

### US-4 普通员工提交提案
> 作为普通员工，我希望能在激励中心提交改善提案，并查看自己的提交记录，但不能使用分享计算器。

**验收标准：**
- 可查看驾驶舱（只读）
- 可提交改善提案
- 可查看积分排行榜
- 分享计算器菜单不显示
- 其他模块菜单不显示

### US-5 角色菜单动态过滤
> 作为任何角色的用户，我希望系统根据我的角色动态展示有权限的菜单项，无权限的模块完全不可见。

**验收标准：**
- 登录后菜单根据角色自动过滤
- 子菜单中无权限的项不显示
- 如果某个子菜单组下所有子项都无权限，整个组不显示
- 切换角色后菜单立即更新

## 六、UI设计要点

### 6.1 用户管理页面（新增）
- 入口：侧边栏新增「系统管理」分组，含「用户管理」子菜单
- 仅 `super_admin` 和 `owner` 可见
- 页面布局：顶部搜索+筛选 → 表格列表 → 弹窗编辑

### 6.2 登录页变更
- 登录成功后，后端返回 `permissions` 数组（可访问的模块代码列表）
- 前端存入 Pinia store，用于菜单过滤

### 6.3 403页面
- 简洁提示：「您没有权限访问该页面」
- 提供「返回首页」按钮

## 七、技术约束

- 后端：Node.js + Express + better-sqlite3
- 前端：Vue 3 + Element Plus
- 权限模型：基于角色的访问控制（RBAC），不做细粒度的权限分配（P0阶段角色权限硬编码）
- 数据库：SQLite，需新增 `roles` 和 `permissions` 表（或用JSON字段存储）
- 兼容性：升级后现有 admin 用户的角色自动映射为 `super_admin`

## 八、待确认问题

1. **是否需要角色管理页面（P1→P0）？** 当前设计P0阶段角色权限硬编码在代码中，如果客户需要灵活调整角色权限则需要提前实现
2. **数据权限是否需要？** 当前P0只做功能权限（菜单/API级别），数据权限（部门级隔离）放P1
3. **admin账号密码策略？** 超级管理员的初始密码是统一分发还是首次登录强制修改？

## 九、非功能性需求

- 权限检查延迟 < 10ms（内存缓存，不每次查DB）
- 角色变更后，已登录用户下次请求即生效（JWT中不含权限详情，每次从DB/缓存读取）
- 并发用户支持：100+（中小企业规模足够）
